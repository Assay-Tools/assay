# State of Agentic Software — Q1 2026

*Published by [Assay](https://assay.tools) — The quality layer for agentic software.*

---

## Executive Summary

The agentic software ecosystem is growing fast — and growing unevenly. As of March 2026, Assay tracks 6,954 packages across 147 categories, from MCP servers and REST APIs to SDKs and agent frameworks. Of those, 2,456 have been evaluated against our five-dimension Agent-Friendliness (AF) scoring rubric. The average AF Score sits at 61.3 out of 100 — solidly "Good" territory, but with enormous variance underneath.

The headline finding: most packages are adequate but few are excellent. Only 2.2% of evaluated packages score 80 or above, while a combined 97% fall between 40 and 79. The ecosystem has cleared the bar of basic functionality — the days of completely broken MCP servers are largely behind us — but the gap between "works" and "works well for agents" remains wide. Documentation accuracy and error message quality continue to be the weakest links for the majority of packages.

The top-scoring packages share a consistent pattern: strong documentation, thoughtful error handling, and simple authentication. Anthropic's Claude API leads at 93.2, followed by Stripe (90.8) and GitHub (89.8) — companies that invested heavily in developer experience long before agents entered the picture. This suggests that agent-friendliness isn't a bolt-on feature; it's the natural result of taking API quality seriously.

## Security Posture

Security in the agentic software ecosystem is a study in contrasts. At the top end, enterprise-grade services like HashiCorp Vault (98.0), Stripe (95.0), and AWS IAM (94.5) demonstrate that it's possible to build highly secure APIs that also work well for agents. These packages enforce TLS, support fine-grained scopes, handle secrets properly, and maintain clean dependency trees.

But the median security score tells a different story. Many packages — particularly community-built MCP servers — treat security as an afterthought. Common patterns include accepting unvalidated input, running with overly broad permissions, and shipping dependencies with known vulnerabilities. For agents operating autonomously, these gaps create real risk: a compromised MCP server can become a vector for data exfiltration or privilege escalation in the agent's execution environment.

### Common Security Gaps

- **Overly broad permissions** — Many packages request or require full account access when narrower scopes would suffice. Agents inherit these permissions, creating unnecessary blast radius.
- **No input validation on tool parameters** — MCP servers that pass user-supplied arguments directly to underlying APIs without sanitization are vulnerable to injection attacks.
- **Stale dependencies** — A significant number of packages ship with outdated dependencies containing known CVEs, particularly in the npm and PyPI ecosystems.
- **Missing TLS enforcement** — Some MCP servers communicate over unencrypted channels in their default configuration, exposing credentials and data in transit.
- **Long-lived API keys with no rotation** — Packages that rely on static API keys with no expiration or rotation mechanism create persistent credential risk.
- **No rate limiting guidance** — Packages that don't document rate limits leave agents to discover them through failures, potentially triggering account lockouts.

## Agent Readiness Patterns

### What Makes a Package Agent-Friendly?

Our analysis of the top-scoring packages reveals five patterns that consistently separate excellent from average:

1. **Documentation matches behavior.** The number-one predictor of a high AF Score is documentation that accurately reflects what the API actually does — not what it was designed to do three versions ago. Agents can't call support; they rely entirely on docs.

2. **Errors tell you what to do, not just what went wrong.** "400 Bad Request" is useless to an agent. "Missing required field `recipient_email` — expected a valid email address" is actionable. The best packages return structured errors with remediation hints.

3. **Authentication is a single step.** API key in a header or environment variable. That's it. Every additional step — OAuth redirect, scope negotiation, token refresh — multiplies the failure surface for autonomous agents.

4. **Tool schemas are complete and typed.** MCP servers with fully specified JSON Schema parameters (including descriptions, defaults, and enums) let agents construct valid calls without guessing. Incomplete schemas force trial-and-error.

5. **Failure modes are documented.** Packages that document their rate limits, known edge cases, and "avoid-when" conditions allow agents to route around problems proactively rather than discovering them at runtime.

## Emerging Trends

**Official MCP servers are arriving.** Q1 saw a notable shift from community-built MCP wrappers to first-party implementations. GitHub, Notion, and several other platforms now ship official MCP servers, and the quality difference is measurable — official servers average 8-12 points higher than their community counterparts on AF Score. This trend validates the MCP protocol as a durable standard and suggests that agent-readiness is becoming a competitive feature, not a community afterthought.

**The "good enough" plateau is real.** A large cluster of packages sit in the 55-70 AF range and show little movement. These packages work, their docs are passable, and their maintainers have moved on to new features rather than polishing the agent experience. Breaking through this plateau typically requires investment in error message quality and documentation accuracy — areas that don't show up in changelogs but matter enormously for agent reliability.

**Security tooling for agents is a fast-growing niche.** The Security category nearly doubled in tracked packages this quarter, driven by MCP servers for reverse engineering (IDA Pro, Ghidra), vulnerability management (Snyk, SonarQube), and identity platforms (Entra ID, AWS IAM). As agents take on more operational tasks — code review, incident response, compliance checks — the quality of security tool integrations becomes critical infrastructure, not a nice-to-have.
